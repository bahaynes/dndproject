# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## ⚠️ Auth is a Protected Area — Read Before Touching

Auth is intentionally simple. **Do not modify auth files unless the task explicitly requires it.**

The full auth contract:
- `localStorage.accessToken` — the only token that matters. Always campaign-scoped.
- `localStorage.pendingToken` + `pendingDiscordToken` — set by `/login/callback`, consumed by `/campaigns`. Nowhere else.
- `frontend/src/lib/auth.ts` — one store, two methods: `login()` and `logout()`. Do not add methods.
- `frontend/src/routes/+layout.svelte` — reads `accessToken`, calls `/api/auth/me`, done. Do not add token types.
- Backend: `get_current_active_user` / `get_current_active_admin_user` in `dependencies.py`. Do not add new auth dependencies.

The flow: Discord OAuth → `/login/callback` (stores tokens, redirects) → `/campaigns` (picks campaign, issues `accessToken`) → everywhere else.

## Working on This Codebase (AI Model Instructions)

**Always work on a branch, never directly on `main`.**

Commit after each logical unit of change — don't batch everything into one commit at the end. This creates rollback points and makes regressions bisectable with `git bisect`. A good rule of thumb: if you've made a change that could independently break something, commit it before moving on.

```bash
git checkout -b your-branch-name   # start on a branch
# ... make a change ...
git add <specific files>
git commit -m "short description"  # commit it immediately
# ... make the next change ...
git commit -m "..."                # commit again
```

**After every commit, smoke tests run automatically** (if the dev pod is up). If they fail, the commit is blocked — fix the issue before proceeding. This catches regressions like broken Vite configs or a crashed frontend before they cost hours of debugging.

If you need to bypass the hook (e.g., you're committing a docs change while the pod is intentionally down): `git commit --no-verify`. Use this sparingly.

**Before touching config files** (`vite.config.ts`, `Containerfile*`, pod YAML, `pyproject.toml`), check the pod logs first:
```bash
podman logs --tail 20 dnd-westmarches-dev-frontend
podman logs --tail 20 dnd-westmarches-dev-backend
```
Config errors show up there immediately and are much faster to diagnose than tracing auth or UI failures.

## Architecture

### Multi-Tenancy
Every user is scoped to a **Campaign** (`users.campaign_id`). JWT tokens encode both `discord_id` and `campaign_id`. All queries must filter by `campaign_id` — never return cross-campaign data.

Two auth dependency levels in `backend/app/dependencies.py`:
- `get_current_active_user` — any active campaign-scoped user
- `get_current_active_admin_user` — DM/admin only (`role == "admin"`)

### Backend Module Pattern
Each feature lives in `backend/app/modules/<name>/` with:
- `models.py` — SQLAlchemy ORM models
- `schemas.py` — Pydantic request/response schemas
- `service.py` — business logic / DB queries
- `router.py` — FastAPI routes

**When adding a new module**, you must import its models in **both**:
1. `backend/app/all_models.py` — so Alembic detects schema changes
2. `backend/app/main.py` — so SQLAlchemy creates tables and the router is registered

All routes are prefixed `/api/<name>` in `main.py`.

### Database
- Dev: SQLite (`sqlite:///./test.db`)
- Prod: PostgreSQL via psycopg 3 driver (`postgresql+psycopg://`)
- `database.py` auto-converts `postgres://` / `postgresql://` URLs
- Tests use an in-memory SQLite DB (`sqlite:///:memory:`) — see `backend/tests/conftest.py`
- SQLAlchemy JSON columns: mutations must replace the whole object, not mutate in-place

### Frontend Patterns
- API calls: use `API_BASE_URL` from `$lib/config` (resolves to `VITE_API_URL` env var or `/api`)
- Auth token: `get(auth).token` from `$lib/auth` store
- Shared types: `frontend/src/lib/types.ts`
- Package manager: **pnpm**; backend uses **uv**

### LLM / One-Shot Generator
`backend/app/llm_service.py` wraps an OpenRouter-compatible API. Configured via `LLM_API_BASE`, `LLM_API_KEY`, `LLM_MODEL` env vars. The oneshot module polls async generation jobs at `/api/oneshot/{id}`.

### Package Managers
- Backend: `uv` (lock file: `uv.lock`)
- Frontend: `pnpm` (lock file: `frontend/pnpm-lock.yaml`)
- Do not switch these without discussion.

## Definition of Done

A feature is not complete until all of the following pass:

1. **Backend unit tests** — `uv run pytest tests/` passes with tests covering the new endpoints/service logic. Add tests to `backend/tests/test_<module>.py`. Use the shared fixtures in `conftest.py` (`campaign`, `player_auth_headers`, `admin_auth_headers`).

2. **Frontend unit tests** — `pnpm test:ci` passes. Add component/page tests in `frontend/src/tests/` or alongside the route using Vitest + `@testing-library/svelte`.

3. **E2E tests** — `./run_e2e.sh` passes. Add a spec in `e2e/tests/` covering the happy path of the feature. Player-facing features go in `player.spec.ts`; admin-facing in `admin.spec.ts`.

4. **All existing tests still pass** — run `./run_tests.sh` to confirm no regressions.

Do not mark a task complete, close a PR, or consider a feature shipped if any of the above are failing or missing.
